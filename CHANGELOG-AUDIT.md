## Lecons-apprises

| Sujet | Lecon |
|-------|-------|
| **Rules sub durcies en Phase A = cout de drift sans gain securite** | Tant que le mono (`/companies/{cid}`) reste AUTORITATIF (Phase A : les listeners lisent le mono, le sub est un mirror write-only fire-and-forget sous session user), durcir field-level les rules des sous-collections ne protege RIEN : le changement financier d un editor atterrit dans le mono que l app lit. Pire, ca BLOQUE des writes legit du mirror : `writeBatch.commit()` est atomique -> un seul item rejete fait echouer TOUTE la batch dep -> mono/sub divergent + `dual_write_errors` permission-denied -> **contamine le signal de la fenetre 7j**. Corollaire structurel : **durcissement sub <=> bascule des reads sur sub**. On bundle O (rules sub) + P (mono field-level) + gating UI au CUTOVER Phase C, jamais en avance. Le 4-eyes utile (s il l etait) vit dans un workflow applicatif, pas dans une rule Firestore. Erreur initiale evitee : j avais plaide "deploy O maintenant, dette secu non-bornee" — disque a tranche, le mono autoritatif rend O Phase-A inutile ET nuisible. |
| **Course `clearFirestore()` scope par projectId (tests rules paralleles)** | Vitest lance les fichiers de test en parallele et `clearFirestore()` est scope par projectId. Deux fichiers rules partageant un projectId -> le `beforeEach` de l un efface les seeds de l autre -> echecs INTERMITTENTS (assertSucceeds qui echoue en PERMISSION_DENIED si le seed membre disparait ; assertFails qui passe si companyA vide fait passer l attaquant pour premier membre). Remede : un projectId distinct par fichier. **Exception** : un fichier dont les rules font un `firestore.get()` cross-service (ex storage-security : membership) DOIT garder le `--project` de l emulateur, car ce get ne resout que contre le projet par defaut de l emulateur, pas le projectId du testEnv. L isolation est alors portee par les AUTRES fichiers. Discipline : isoler la racine (run en isolation) plutot que re-run jusqu au vert. |
| **Backlog P (mono field-level) intractable par les rules = subsume par la migration sub** | P tel que concu (etendre `changesAdminOnlyFields` aux financiers sur le mono `/companies/{cid}`) n'est PAS realisable. Les financiers (`rv,pf,gar_*,mt,ht,pc`) ne sont pas top-level sur mono : ils vivent DANS les elements d'arrays (`dos:[{id,rv,pf,...}]`). Or (a) `affectedKeys()` ne voit que les clefs top-level -> un changement de `dos[3].rv` ne remonte que `'dos'`, jamais quel element ni quel champ ; (b) `diff()` n'est defini que sur Map, PAS sur List -> aucune primitive pour differ un array element par element ; (c) pas d'iteration (`forEach`/`every`) dans le rule language ; (d) `toSet()` est tout-ou-rien (trop strict). Le blocage est le MODELE DE DONNEES (array imbrique), pas le verbe d'ecriture : un `updateDoc` dot-path n'aide que pour les maps imbriquees, jamais les listes. Reframe durable : P n'est pas "infaisable" mais SUBSUME par la migration sub. Le seul modele qui rend le field-level mono tractable = doc-par-item keye par id = exactement ce qu'EST le sub. Donc **P == Phase C** ; coder P separement = recoder Phase C sur le mono (redondant par construction). Elimine du backlog. Le vrai "durcissement mono" est sa DISPARITION en Phase D. |
| **"Mono corrompu = aucun impact applicatif" est une propriete de Phase D, pas de Phase C (prev-sourcing load-bearing)** | Apres la bascule des reads vers sub (Phase C), le mono n'est plus lu par l'UI MAIS reste lu par `resolvePrevSnapshot` (backlog G) : `save()` fait un `getDoc` du mono pour sourcer `prev` et calculer le diff du dual-write vers sub. Scenario concret : un editor ecrit directement le mono via SDK (hors `save`) -> mono `[A,B]` (C supprime), sub intact `[A,B,C]`. Un delete legitime de C ensuite : `prev` lu du mono corrompu = `[A,B]` (C deja absent) -> le Step 2 du mirror cherche les ids de `prev` absents de `newData` -> ne trouve pas C -> **C n'est PAS supprime du sub** -> drift permanent, intention utilisateur masquee. Et puisque l'app lit sub (Phase C+), C reapparait dans la React state au prochain push onSnapshot -> l'utilisateur voit C revenir apres l'avoir supprime (symptome support = "j'ai supprime, ca revient") = bug visible de la pire classe (action qui ne tient pas), pas un simple drift technique. Tant que `prev` source du mono, un mono corrompu se propage/se masque vers sub. Donc le scope Phase C inclut NON SEULEMENT la bascule des listeners mais AUSSI le deplacement du prev-sourcing mono->sub. Options a designer au chantier (non tranchees) : (1) `getDocs` par sous-collection au save (authoritative, N reads/save, OK au volume Sapurai) ; (2) reutiliser `dataRef` (G+N) comme source de verite locale du prev (zero reseau, mais dataRef devient autoritatif) ; (3) cache local de prev tenu a jour par les listeners eux-memes (le plus propre : "ce que je viens de voir sub = mon prev"). |
